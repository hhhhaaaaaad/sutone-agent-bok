## 四、核心工作与产出（按亮点排序）

### Agent 差异化线（招牌，放前面）

> **1. 多 Agent 写作流水线**：基于 Google ADK 设计「规划→写作→配图→质检」四阶段串行工作流，消费 ADK 事件流时按 `author` 区分 Agent 阶段，仅将审查 Agent 的终稿落库，解决了多 Agent 输出相互污染导致最终结果混杂的问题。

> **2. 工作流复用与嵌套**：将 draw.io 绘图的三 Agent 子工作流（需求分析→绘图→布局审查）作为可复用子流程，嵌入写作流水线的配图环节，实现文章自动配图；同一套绘图编排同时服务独立画图页与文章配图两个场景。

> **3. RAG 检索增强**：将用户历史文章切分、embedding 后存入 pgvector，写作前检索 topK 相关片段注入 prompt；实现「固定前置检索」与「封装为 MCP 工具由 Agent 自主调用（Agentic RAG）」两种范式，显著降低事实性幻觉。

> **4. Agent 工具调用可视化**：将 Agent 每步工具调用（联网检索/RAG）的参数、返回、耗时通过 SSE 实时推送前端展示，兼作调试手段与用户可解释性。

### Java 后端工程线

> **5. 认证与安全**：基于 Spring Security 过滤器链 + JWT 无状态鉴权（access+refresh 双 token 续签），密码 BCrypt 加盐，接口做资源归属校验防水平越权。

> **6. 中间件与性能**：Redis 落地 AI 接口令牌桶限流（控制 LLM 调用成本）、文章缓存（处理穿透/击穿/雪崩）、AI 任务提交分布式锁防重复；文章表联合索引 + 深分页游标改写，核心接口单机压测 QPS 达 **XXX**（JMeter，待实测填）。

> **7. 可观测与部署**：MDC + traceId 打通「一次 AI 请求→四个 Agent 阶段」全链路日志；Docker Compose 编排 Nginx+前后端+MySQL+Redis+pgvector 部署至云服务器，HTTPS + API Key 环境变量隔离，公网可演示。

## 五、消息队列（MQ）应该用在什么场景 + 怎么写进简历

> 这是你专门问的问题。MQ 不能「为了用而用」，简历上要写出**有真实业务动机**的场景，否则面试官一句「你这为什么要用 MQ、不用行不行」就把你问住。下面三个场景都是你项目里能自然长出来的。

### 5.1 三个「有动机」的落地场景（按推荐度排序）

| # | 场景 | 为什么必须异步（动机） | MQ 解决的核心问题 | 面试考点 |
|---|------|----------------------|------------------|---------|
| **① AI 写作任务异步化**（首推） | 提交任务 → 进 MQ → 消费者调 Agent 生成 → 结果经 SSE/WebSocket 推回 | LLM 调用耗时长（几秒~几十秒），同步阻塞会长时间占用 HTTP 连接、扛不住并发提交 | **削峰填谷**、异步解耦、任务状态机（PENDING→RUNNING→SUCCESS/FAILED） | 消息可靠投递、消费幂等、失败重试与死信队列 |
| **② 文章发布后的异步统计/索引** | 发布成功后 → 发消息 → 异步更新阅读量、作者文章计数、（未来）搜索索引、RAG 向量库 | 这些是「发布」的副作用，不应阻塞主流程、也不该让用户等 | **主流程与副作用解耦**，一条消息多个消费者各干各的 | 发布订阅模型、最终一致性、多消费组 |
| **③ AI 任务结果通知** | 任务完成 → 发消息 → 通知服务推送给用户（站内信/前端） | 生成可能很慢，用户可能已离开页面，需异步通知 | 异步通知、离线消息 | 消息顺序、消费确认 |

> **强烈建议主打场景 ①**：因为「LLM 调用慢 + 贵」是你项目独有的、真实的异步动机，比通用的「秒杀削峰」更贴合你项目、更好讲。

### 5.2 技术选型建议
- **RocketMQ**（推荐）：国内大厂用得多、面试认可度高，事务消息/顺序消息/死信队列特性全，适合讲「可靠投递」。
- RabbitMQ：轻量、上手快，适合场景 ②/③。
- Kafka：更偏日志/大数据流，你这个业务体量用它「杀鸡用牛刀」，除非想讲高吞吐，否则不必。

> 校招选 **RocketMQ** 讲解性价比最高。

### 5.3 结合 AI 任务异步化的完整链路（面试要能画出来）
```text
用户提交写作任务
  → Controller 落库 PENDING 任务，发送消息到 MQ（携带 taskId），立即返回 taskId（不阻塞）
  → 消费者拉取消息 → 查任务 → 调多 Agent 流水线生成
  → 生成中：更新任务状态 RUNNING，token 经 SSE/WebSocket 推前端
  → 成功：状态 SUCCESS + 结果落库；失败：重试 N 次仍失败 → 进死信队列 + 状态 FAILED
  → 前端凭 taskId 订阅结果 / 轮询状态
```

**幂等设计**（面试必问「消息重复消费怎么办」）：
> 消费前先查任务状态，若已是 SUCCESS/RUNNING 则直接丢弃（或用 taskId 做去重键存 Redis），保证同一任务不被重复生成——这既省 token 又保证幂等。

### 5.4 写进简历的条目（【待落地】完成后再放正式简历）

**标准版（一条，推荐）：**
> **8.【待落地】AI 任务异步化**：针对 LLM 调用耗时长、同步阻塞占用连接的问题，用 RocketMQ 将 AI 写作任务异步化——提交即入队并返回 taskId，消费者异步执行多 Agent 流水线，结果经 SSE 推送；通过消费幂等（Redis 去重键）+ 死信队列保证不重复生成、失败可追溯。

**精简版（并进后端条目里）：**
> 用 RocketMQ 将耗时的 AI 生成任务异步化，削峰填谷并解耦主流程，配合消费幂等与死信队列保证可靠性。

**若同时做了场景 ②，可补一句：**
> 文章发布后通过 MQ 发布订阅异步更新统计计数与检索索引，主流程与副作用解耦，保证最终一致性。

### 5.5 MQ 相关面试话术与追问准备
> 「AI 写作任务用 RocketMQ 异步化。因为 LLM 调用又慢又贵，同步 stream 会长时间占用连接、并发一高就扛不住。改成提交即入队返回 taskId，消费者异步跑多 Agent 流水线，结果经 SSE 推回。消费幂等用 taskId 做 Redis 去重键，重试多次仍失败进死信队列并标记任务 FAILED。」

> 延伸追问准备：
> - **为什么用 MQ 而不是线程池异步？** → 线程池是单机内存态，重启任务丢失、无法削峰、无法多实例分摊；MQ 持久化、可重试、天然支持水平扩展消费。
> - **消息丢失怎么办？** → 生产者确认（同步发送/事务消息）、Broker 持久化、消费者手动 ACK 三段保证。
> - **消息重复怎么办？** → 消费幂等（taskId 去重键 / 先查状态再执行）。
> - **消息堆积怎么办？** → 扩容消费者、监控堆积量告警、必要时降级。
> - **为什么选 RocketMQ 不选 Kafka？** → 业务是任务型消息、需要事务消息和死信队列，RocketMQ 更贴合；Kafka 更适合高吞吐日志流，这里用不上它的强项。

> **诚实提醒**：MQ 在整体方案里是 P1（阶段八），优先级低于 Agent 主线和用户体系。**没做完就别写进正式简历正文**，可在面试口头讲「异步化我的设计方案是…」。做完场景 ① 一个就够写、够讲，不必三个都堆。
