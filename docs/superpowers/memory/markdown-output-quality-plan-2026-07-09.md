# Markdown 输出质量优化方案 — 2026-07-09

> 问题：`agent-writing.yml` 的 prompt 约束 + `AiWritingService.formatMarkdown` 的 20+ 条正则，仍无法稳定产出规范 Markdown。
> 本文档给出**根因诊断 + 分层优化方案 + 落地步骤 + 面试话术**。

---

## 1. 根因诊断（附代码级铁证）

案例里的畸形分两类，**一半是 formatMarkdown 自己制造的**：

### A 类：正则「误切」制造的 bug（formatMarkdown 主动搞坏）
- `## 六、实战：\n\n通过 JVM参数调优内存`：完整标题被拆成「标题+段落」。
  - 元凶：规则 11A `^(#{1,6}\s.{1,40}[。：])([^\n]{10,})$` —— 见标题含冒号就在冒号后换行。已本地复现。
- 连锁误伤：`### 6.1典型场景...示例1. **高并发...` 等。

### B 类：正则「漏切」没覆盖（LLM 畸形，规则抓不住）
- `### 4.2 NIO中的 Buffer与 Channel如何利用直接内存通过 ...`：标题正文粘连。无标点、正文不以 `**` 开头 → 11A/11B 特征都不匹配，永久漏切。已复现「不能切」。
- `### 2.2程序计数器`、`### 4.3 ...参数配置-用于限制`：序号/破折号粘连，同类漏网。
- `\*\*JVM运行时数据区\*\*`：转义残留。

### C 类：结构性问题（流式与格式化脱节）
- `formatMarkdown` 只在最后 `result` 事件跑一次（`AiWritingService.java:115-117`）。
- 流式 `token` 推给前端的是**原始未格式化内容**（前端 `AiWritingPanel/index.tsx:124` 累加 raw token）。
- 结果：流式全程看到畸形，最后一刻才被 result 覆盖。用户发的案例多半是这个中间态。

---

## 2. 为什么正则后处理走不通（方向性结论）

| 死结 | 说明 |
|------|------|
| 有限规则 vs 无限畸形 | LLM 畸形样式无穷，正则永远补不完 |
| 规则互相打架 | 「切粘连」的规则会「误切合法标题」，补一洞炸一洞（A 类 bug 来源） |
| 流式天生缺上下文 | 判断「标题后要不要空行」需看下一行，流式时下一行还没到 |

**结论：不是加正则能解决的，要换方向——用「真正的 Markdown 解析器（AST）」或「结构化输出」替代字符串正则。**

## 3. 分层优化方案（从低成本到高收益，可叠加）

### 层一：先止血——删掉「误切」正则，救回被 formatMarkdown 搞坏的部分（0.5 天，立即见效）

A 类 bug 是**你自己的正则制造的**，删掉比留着强：
- 删/收紧规则 11A（`[。：]` 后强切）、11B（标题后 `**` 强切）——它们误伤率 > 命中率。
- 保留真正安全的：4c 反转义 `\*`→`*`、5/6/8 标题空行、14 列表空格、17/19/20 表格修复。
- **判断标准**：一条规则若可能改动「本来就合法」的文本，就是危险规则，删。

> 收益：立刻消除 `## 六、实战：\n\n通过...` 这类项目自造畸形。

### 层二：用 AST 解析器替代正则做规范化（2-3 天，根治 A/B 类）★ 推荐主方案

引入 **CommonMark-Java（commonmark + gfm 扩展）**，把「正则改字符串」换成「解析成 AST 再重新渲染」：

```text
LLM 原始输出 → CommonMark 解析成 AST → 遍历修正(合并/补空行/规范表格) → 重新序列化为标准 Markdown
```

- 解析器天然理解「什么是标题、什么是段落、什么是表格」，**不会误切合法标题**（根治 A 类）。
- 重新序列化时统一补空行、统一层级，**对无穷畸形样式鲁棒**（缓解 B 类）。
- pom 需新增依赖（当前项目无 markdown 解析库）：
  ```xml
  <dependency>
    <groupId>org.commonmark</groupId><artifactId>commonmark</artifactId><version>0.22.0</version>
  </dependency>
  <dependency>
    <groupId>org.commonmark</groupId><artifactId>commonmark-ext-gfm-tables</artifactId><version>0.22.0</version>
  </dependency>
  ```
- `formatMarkdown` 从「20+ 条正则」重写为「parse → 轻量 AST 修正 → render」，只对**解析器搞不定的少数畸形**（如标题正文粘连）保留 2-3 条预处理正则。

> **注意**：AST 化能根治「结构错乱」，但 B 类里「标题和正文粘在同一行且无任何分隔特征」这种，解析器也会当成一个整标题——因为它本就无法从文本推断边界。这类必须靠层三/层四从源头减少。

### 层三：从源头治理——改 prompt 策略 + 用 few-shot（1-2 天，减少畸形产生）

现在 `agent_writing_generator` 的 instruction 堆了 30+ 条格式规则（`agent-writing.yml`），**规则越多，模型越容易顾此失彼**。优化：
1. **精简规则**：只留最关键的 5-8 条硬规则，删掉互相冲突/过度细节的（如各种 emoji 提示块、勾选清单）。
2. **加 few-shot 示例**：给 1-2 个「标准格式的输入输出范例」，比 30 条文字规则有效得多——模型擅长模仿，不擅长遵守长清单。
3. **让 reviewer 专职格式**：现在 reviewer 既查技术又查格式，注意力分散。可让 reviewer 的 instruction 聚焦「只修 Markdown 格式规范」，技术准确性交给 generator。

### 层四：结构化输出（3-5 天，终极方案，面试最亮）★★

不再让 LLM 直接吐 Markdown 字符串，而是**吐结构化 JSON，后端渲染成 Markdown**——这正是你 draw.io/PPT 已经在用的思路（`{"type":"drawio_node",...}` 按行输出、后端渲染）！

```json
{"type":"heading","level":2,"text":"线程私有内存区域"}
{"type":"paragraph","text":"程序计数器是..."}
{"type":"table","headers":["区域","异常"],"rows":[["堆","OOM"]]}
{"type":"code","lang":"java","text":"..."}
```

- 后端按 type 拼装标准 Markdown，**格式 100% 可控**（空行、层级、表格对齐由代码保证，LLM 再乱也不影响结构）。
- 与 draw.io 一致的「Agent 输出结构化数据 + 后端/前端渲染」范式，**架构统一**。
- 面试点极强：「我把易错的自由文本生成，转成了结构化生成 + 确定性渲染，把格式正确性从『依赖模型自觉』变成『代码保证』。」

## 4. 还要修一个结构性问题：流式与格式化脱节（层一起就该带上）

**现状**（`AiWritingService.java`）：`token` 事件推原始内容，只有终末 `result` 事件跑 `formatMarkdown`。用户流式全程看到畸形。

**两种修法**：
- **简单**：前端流式阶段标注「生成中，格式化后为准」，收到 `result` 用格式化终稿替换预览（前端已有 `result` 分支 `AiWritingPanel/index.tsx:125`，语义已具备，补个「预览态」提示即可）。
- **彻底**：结合阶段〇的 author 感知——generator 阶段的 token 明确标为「初稿预览」，reviewer 的终稿才 `formatMarkdown` 后作为 result。与 `final_plan` 阶段〇天然合并做。

## 5. 落地步骤与优先级

| 步骤 | 动作 | 成本 | 收益 | 建议 |
|------|------|------|------|------|
| 1 | 层一：删危险正则（11A/11B 等误切规则） | 0.5 天 | 立即消除自造畸形 | **马上做** |
| 2 | 层三：精简 prompt + 加 few-shot | 1-2 天 | 减少畸形产生 | **马上做** |
| 3 | 层二：CommonMark AST 重写 formatMarkdown | 2-3 天 | 根治 A 类、大幅缓解 B 类 | **主方案** |
| 4 | 层四：结构化输出 + 后端渲染 | 3-5 天 | 格式 100% 可控 | 时间够就做，面试最亮 |
| 5 | 修流式脱节（并入 final_plan 阶段〇） | 含在阶段〇 | 流式全程不畸形 | 跟阶段〇一起 |

**推荐路径**：先做 1+2（半天见效止血）→ 再做 3（根治）→ 行有余力做 4（终极 + 面试亮点）。

## 6. 面试话术

> 「AI 生成 Markdown 最初我用正则后处理，但发现两个问题：一是正则规则会互相冲突，修畸形的规则会误伤合法标题；二是 LLM 的畸形样式无穷，正则永远补不完。所以我分三层重构：源头上精简 prompt 并加 few-shot 示例；处理上用 CommonMark 解析成 AST 再重新渲染，从根本上避免误切;最彻底的是把自由文本生成改成结构化 JSON 输出、后端确定性渲染——这和我 draw.io 链路的思路一致，把格式正确性从『依赖模型自觉』变成『代码保证』。」

> 追问准备：
> - 为什么不继续加正则？→ 正则无法理解语义，改字符串会误伤合法内容，且畸形无穷；AST 理解结构，从根上避免。
> - 结构化输出的代价？→ 牺牲一点生成灵活性、prompt 要求更严，但换来格式确定性，对「要入库要渲染」的场景值得。
