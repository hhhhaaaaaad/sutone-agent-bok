## 文章回退草稿编辑方案

### 核心思路
新增后端接口 `POST /api/v1/articles/{articleId}/revert-to-draft`，将已发布文章关联的草稿重新置为可编辑状态。前端文章详情页增加"继续编辑"按钮，调用该接口后跳转到草稿编辑器。

### 涉及修改的文件

1. **`src/api/articles.ts`** — 新增 `revertToDraft` 方法
2. **`src/app/articles/[articleId]/page.tsx`** — 新增"继续编辑"按钮 + 调用逻辑
3. **后端服务** — 新增 `POST /api/v1/articles/{id}/revert-to-draft` 接口

### 后端接口约定

| 项目 | 内容 |
|---|---|
| 方法 | `POST` |
| 端点 | `/api/v1/articles/{articleId}/revert-to-draft` |
| 请求体 | 无 |
| 成功响应 | `{ code: "0000", info: "success", data: { draftId: number } }` |
| 后端行为 | 将原草稿 status 重置为 0（可编辑）；可选同时下架文章 |

### 前端修改细节

**`src/api/articles.ts`**：新增 `revertToDraft` 方法，与现有 API 风格一致。

**`src/app/articles/[articleId]/page.tsx`**：
- 新增 `reverting` 状态和 `handleRevertToDraft` 处理函数
- 在 header 区域"返回文章广场"按钮旁添加"继续编辑"按钮
- 调用成功后跳转 `/drafts/{draftId}`，失败时显示错误

### 注意事项
- 此方案依赖后端新增接口，如果后端短期内无法配合，替代方案是前端通过 `draftsApi.save()` 创建新草稿并拷贝文章内容，但不推荐（失去关联关系、数据冗余）
