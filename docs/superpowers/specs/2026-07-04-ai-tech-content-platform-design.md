# AI 技术知识生产与社区平台设计稿

## 1. 文档背景

当前项目原本以 `Agent 工作台 + Draw.io/PPT 工具能力` 为主，更接近原型验证形态。现阶段产品方向已明确调整为：

- 主定位：`AI 技术知识生产与社区平台`
- 落地节奏：`路线 C 定位 + 路线 A 节奏`
- 实施策略：先完成 `AI 创作发布闭环`，再逐步扩展 `技术社区互动能力`

说明：

- 当前工作台界面仅作为平台中的一个工具/能力入口，不再是项目的主要需求场景。
- `Draw.io` 暂不作为第一阶段核心模块，后续以创作增强插件的方式接入。

## 2. 项目定位

项目目标不是单纯做一个博客网站，也不是单纯做一个 AI 工具站，而是围绕技术内容生产构建完整平台，形成如下主链路：

`选题 -> AI 辅助创作 -> 草稿编辑 -> 内容发布 -> 社区传播 -> 知识沉淀`

建议项目名称表达：

- `AI 技术知识生产与社区平台`
- `面向技术内容生产的 AI 创作与社区平台`

## 3. 分阶段建设思路

### 3.1 子项目 A：AI 创作发布中心（第一阶段）

目标：优先跑通“技术内容从创作到发布”的最小完整闭环。

核心能力：

- AI 文章生成
- AI 润色、续写、改写、总结
- Markdown 编辑与预览
- 草稿保存与编辑
- 标签、摘要、封面等发布信息维护
- 发布文章与文章详情展示

### 3.2 子项目 B：技术社区互动中心（第二阶段）

目标：补齐平台的社区属性，使其从创作平台升级为内容社区。

核心能力：

- 首页推荐流
- 点赞、收藏、评论
- 标签、分类、专栏
- 作者主页、关注关系
- 热门文章与基础推荐排序

### 3.3 子项目 C：知识资产与平台治理中心（第三阶段）

目标：增强平台的长期沉淀价值和工程治理能力。

核心能力：

- 文章版本管理
- 内容资产管理
- Agent 配置中心
- 模型配置中心
- 内容审核/发布治理
- 数据统计与运营分析

## 4. 核心产品模块地图

### 4.1 前台模块

1. `首页`
   - 推荐文章、最新文章、热门标签、精选作者
   - 第一阶段可先做轻量分发能力

2. `AI 创作中心`
   - 第一阶段核心模块
   - 支持选题生成、大纲生成、正文生成、润色改写、标题优化、摘要生成、标签建议、封面建议

3. `文章发布页 / 文章详情页`
   - 发布页负责发布前确认与元信息维护
   - 详情页负责文章展示与后续互动承载

4. `个人中心`
   - 第一阶段优先支持“我的草稿”“我的文章”
   - 后续扩展收藏、互动记录、关注管理

5. `社区互动页`
   - 第二阶段重点建设
   - 包括评论、点赞、收藏、关注、热度展示等能力

### 4.2 后台支撑模块

6. `内容管理后台`
   - 草稿、文章、分类标签、发布状态管理

7. `AI 能力配置后台`
   - 模型配置、Prompt 配置、Agent 配置、生成策略配置

## 5. 第一阶段功能闭环

第一阶段建议只围绕最有价值的创作发布闭环展开，不追求一次性补齐社区全量能力。

### 5.1 闭环功能

1. `用户登录 + 身份体系`
2. `AI 创作工作台`
3. `草稿箱与内容编辑`
4. `发布流程`
5. `文章详情展示页`
6. `个人中心`

### 5.2 核心用户链路

`登录 -> AI 创作 -> 草稿编辑 -> 发布 -> 文章展示 -> 回到个人中心管理`

### 5.3 第一阶段暂缓项

以下能力先预留设计，不作为第一阶段核心交付：

- 评论
- 点赞
- 收藏
- 推荐算法
- 关注关系
- Draw.io 深度集成

## 6. 核心数据对象与业务模型

第一阶段优先围绕“创作态、任务态、发布态”建立业务模型。

### 6.1 User

平台用户主体，承载：

- 注册登录
- 基础资料
- 作者身份
- 个人中心内容归属

### 6.2 Draft

创作草稿对象，代表尚未发布、可持续编辑的内容态。

建议承载信息：

- 标题
- 摘要
- Markdown 正文
- 标签/分类草稿
- 封面草稿
- 当前编辑状态
- 最近一次 AI 生成结果

### 6.3 Article

已发布文章对象，代表正式对外可见的发布态内容。

建模建议：

- `Draft` 面向创作过程
- `Article` 面向发布结果

### 6.4 AiTask

一次 AI 内容生成/改写请求的业务任务对象。

第一阶段可覆盖：

- 生成标题
- 生成大纲
- 生成正文
- 润色改写
- 生成摘要
- 生成标签

后续可扩展：

- 任务状态
- 失败重试
- 消耗统计
- 生成审计

### 6.5 ArticleMeta

文章发布元信息视角，用于管理：

- 标签
- 分类
- 封面
- 字数
- 发布状态

说明：

- 第一阶段不强制要求独立建表
- 但在建模思路上，应与正文内容区分开

### 6.6 业务关系

- 一个 `User` 可以拥有多个 `Draft`
- 一个 `Draft` 可以关联多次 `AiTask`
- 一个 `Draft` 最终可以发布为一个 `Article`
- 一个 `Article` 对应一套发布元信息

## 7. 后端架构演进方案

### 7.1 总体原则

保留当前多模块 Maven 工程结构，不建议推翻重来，而是顺着现有脚手架演进：

- `api`
- `app`
- `domain`
- `trigger`
- `infrastructure`
- `types`

架构重心从：

- `以 Agent 对话为中心`

迁移为：

- `以内容创作发布业务为中心，Agent 作为底层能力`

### 7.2 建议的领域拆分

### account

负责用户注册、登录、资料维护等用户域能力。

### content

负责草稿、文章、内容编辑与管理能力。

### ai-writing

负责 AI 创作任务编排，不再直接以“聊天接口”为业务主入口。

### publishing

负责草稿转文章、内容发布、状态流转等发布链路。

### 7.3 各层职责建议

### trigger

对外 HTTP 接口层，后续建议拆分为：

- 用户接口
- 创作接口
- 草稿接口
- 发布接口
- 文章接口

### domain

承载核心业务规则：

- 草稿创建/更新逻辑
- 发布校验逻辑
- AI 创作任务业务编排
- 文章状态流转

### infrastructure

承载基础设施实现：

- MySQL 持久化
- Redis 缓存
- LLM/Agent 调用适配
- 文件/封面存储
- Prompt 模板加载

### app

保留为启动装配层，负责：

- Spring Boot 启动
- Bean 装配
- 配置加载
- Agent/Plugin 注册

### 7.4 对现有 ChatService 的演进建议

当前项目中 `ChatService` 以“会话 + 消息”为中心，更适合原型阶段。

后续第一阶段建议逐步弱化 `Chat` 作为业务主模型的地位，转向：

- `AiTaskService`
- `AiWritingService`
- `AiContentGenerationService`

即：

- `Chat` 不再是业务主模型
- `AiTask` 才是业务主模型
- Agent 的职责是完成内容生成任务

### 7.5 推荐后端主链路

建议的第一阶段后端调用链为：

`Controller -> 领域服务 -> AI任务执行 -> 草稿落库 -> 发布服务 -> 文章落库`

相比简单的：

`Controller -> Chat -> 返回文本`

上述链路更适合支撑平台化内容生产。

## 8. Draw.io 的当前定位

`Draw.io` 在第一阶段不进入核心业务闭环，当前仅保留为后续增强能力。

后续可作为 `AI 创作中心` 的扩展插件接入，例如：

- 插入技术图解
- 根据文章自动生成配图建议
- 图文联合编辑
- 文章内嵌流程图、架构图、时序图

因此它在平台中的定位应为：

`创作增强器，而不是主产品本身`

## 9. 当前阶段的设计结论

当前已确认的核心决策如下：

- 采用 `路线 C 定位 + 路线 A 落地节奏`
- 先做 `AI 创作发布中心`
- 后续补齐 `社区互动中心`
- 最后补 `知识资产与平台治理中心`
- 第一阶段暂不把 `Draw.io` 纳入核心主流程
- 后端架构从“对话中心”转向“内容生产中心”
