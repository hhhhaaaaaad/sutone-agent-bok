## 一、为什么 Agent 需要记忆系统？

### 1.1 没有记忆的 Agent 是什么样的

想象你每天和同一个客服聊天，但他永远不记得你昨天说了什么：

```
第 1 天：
用户："我是后端工程师，用 Java，帮我写篇 JVM 调优文章"
Agent："好的，这是一篇 JVM 调优文章..."

第 2 天：
用户："帮我写篇关于线程池的文章"
Agent："好的，你是什么背景？用什么语言？" ← 忘记了你是 Java 工程师
```

这就是**无记忆 Agent**的问题。每次对话都是一张白纸。

### 1.2 有记忆的 Agent 能做什么

```
第 2 天：
用户："帮我写篇关于线程池的文章"
Agent：（内部检索到记忆：用户是 Java 后端、喜欢表格对比、标题带编号）
Agent："这是一篇 Java 线程池的技术文章，用了你喜欢的表格对比格式..."
```

**记忆系统的本质**：让 Agent 具备跨会话的「记住用户是谁、用户喜欢什么」的能力。

### 1.3 记忆 vs 聊天历史

| 维度 | 聊天历史（Chat History） | 记忆系统（Memory） |
|------|--------------------------|-------------------|
| 存储内容 | 原始对话消息（user/assistant 来回） | 从对话中提炼的结构化信息 |
| 生命周期 | 单次会话内 | 跨会话持久化 |
| 检索方式 | 按时间顺序翻页 | 按语义相似度搜索 |
| 数据量 | 可能几万条消息 | 精炼后几十~几百条记忆 |
| 类比 | 你和朋友的所有聊天记录 | 你脑中对这个朋友的印象和了解 |

## 三、模块 1：LLM 记忆抽取（Memory Extraction）

### 3.1 这个模块做什么

从一段对话中，用 LLM「读懂」并「提炼」出值得长期记住的信息。

**输入**：一段对话消息
```
用户："我最近在学 Rust，之前一直写 Java，喜欢用 IntelliJ IDEA"
```

**输出**：结构化的记忆列表
```json
{"memory": [
  {"text": "正在学习 Rust", "attributed_to": "user"},
  {"text": "之前一直使用 Java", "attributed_to": "user"},
  {"text": "偏好使用 IntelliJ IDEA", "attributed_to": "user"}
]}
```

### 3.2 它是怎么做的

**核心方法：Prompt Engineering + JSON Structured Output**

Mem0 设计了一个精心构造的 prompt，告诉 LLM：
1. 你要提取哪些类型的信息（偏好、个人信息、计划、健康、职业等）
2. 哪些信息不要提取（闲聊、系统消息、无意义内容）
3. 输出必须是 JSON 格式
4. 提供 few-shot 示例（给 LLM 提供少量示例，通常是几个）让 LLM 理解

Mem0 源码中有三个版本的抽取 prompt：

| Prompt | 用途 | 源码位置 |
|--------|------|----------|
| `FACT_RETRIEVAL_PROMPT` | 早期版本，从整段对话提取 | `configs/prompts.py:15` |
| `USER_MEMORY_EXTRACTION_PROMPT` | 只从 user 消息提取 | `configs/prompts.py:63` |
| `ADDITIVE_EXTRACTION_PROMPT` | V3 最新版，支持去重和链接 | `configs/prompts.py:468` |

### 3.3 V3 版本的关键改进：传入已有记忆做去重

V3 的 prompt 中会包含「已有记忆」，让 LLM 在抽取时就知道哪些已经记住了：

```
## Existing Memories（已有记忆，不要重复提取）
[{"id": "0", "text": "用户是 Java 工程师"}, ...]

## New Messages（新消息，从这里提取）
用户："我用 Java 17 + Spring Boot 3.4"

→ LLM 判断："Java 工程师"已有，但"Java 17 + Spring Boot 3.4"是新的细节
→ 输出：{"memory": [{"text": "技术栈 Java 17 + Spring Boot 3.4"}]}
```

这就是 Mem0 三层去重中的**第一层：LLM 语义感知去重**。

### 3.4 为什么不用规则匹配代替 LLM

- 规则无法理解语义："我不喜欢写很长的文章" → 需要理解这是一个偏好
- 规则无法处理模糊表达："之前那种格式就行" → 需要结合上下文理解
- 规则无法判断价值："你好" → 无记忆价值，"我是 Java 工程师" → 有价值

LLM 是唯一能在语义层面判断「什么值得记住」的工具。

## 四、模块 2：Embedding 向量化

### 4.1 什么是 Embedding

Embedding 就是把一段文字变成一组数字（向量），使得**语义相似的文字在数字空间中距离更近**。

```
"Java 线程池"  → [0.12, -0.45, 0.78, ..., 0.33]   (1024 个浮点数)
"JVM 并发编程" → [0.14, -0.42, 0.75, ..., 0.31]   (非常接近！)
"巧克力蛋糕"   → [-0.67, 0.23, -0.11, ..., 0.89]  (完全不同)
```

### 4.2 为什么需要 Embedding

因为传统的关键词匹配无法处理**同义不同词**的情况：

| 查询 | 关键词匹配 | 语义匹配（Embedding） |
|------|-----------|---------------------|
| "JVM 调优" | 只能匹配包含"JVM"或"调优"的记忆 | 能匹配到"Java 虚拟机性能优化"、"GC 参数配置" |
| "写作风格" | 只能匹配包含"写作"或"风格"的记忆 | 能匹配到"喜欢用表格对比"、"标题带编号" |

### 4.3 Mem0 中 Embedding 的使用场景

1. **写入时**：将每条提取出的记忆文本 embed 成向量，存入向量库
2. **检索时**：将用户查询 embed 成向量，与向量库中的所有向量计算相似度
3. **去重时**：将新消息 embed 后，搜索已有记忆中是否有相似的

### 4.4 余弦相似度（Cosine Similarity）

衡量两个向量"方向是否一致"的方法：

```
cosine(A, B) = (A·B) / (|A| × |B|)

结果范围：[-1, 1]
- 1.0 = 完全相同方向（语义完全一致）
- 0.0 = 正交（完全无关）
- -1.0 = 完全相反（语义对立）
```

实际应用中，cosine > 0.9 一般认为是语义等价，cosine > 0.7 认为是相关。

## 五、模块 3：Vector Store 向量存储与检索

### 5.1 向量库是什么

向量库就是一个专门存储和检索向量的数据库。普通数据库用 SQL 查询，向量库用**相似度搜索**。

```
传统数据库：SELECT * FROM users WHERE name = 'John'     ← 精确匹配
向量库：     SEARCH vectors WHERE cosine(query, vec) > 0.7 ← 近似匹配
```

### 5.2 Mem0 支持的向量库

Mem0 通过工厂模式支持 24 种向量库（`mem0/utils/factory.py`）：

| 向量库 | 特点 | 适用场景 |
|--------|------|----------|
| Qdrant | 高性能、支持过滤 | 生产环境首选 |
| pgvector | PostgreSQL 扩展 | 已有 PG 的项目 |
| Milvus | 分布式、大规模 | 十亿级向量 |
| Chroma | 轻量、嵌入式 | 原型开发 |
| FAISS | Facebook 开源、纯内存 | 低延迟场景 |

### 5.3 向量库的核心操作

```python
# 写入：存储一条记忆的向量
vector_store.insert(
    vectors=[embedding],        # 向量
    ids=["memory-uuid"],        # 唯一ID
    payloads=[{"data": "用户是Java工程师", "hash": "abc123"}]  # 元数据
)

# 搜索：找到最相似的 K 条记忆
results = vector_store.search(
    query="Java开发",           # 查询文本
    vectors=query_embedding,    # 查询向量
    top_k=10,                   # 返回前 10 条
    filters={"user_id": "123"} # 过滤条件
)
```

### 5.4 为什么不用 MySQL 的 LIKE 代替向量库

| 维度 | MySQL LIKE | 向量库 |
|------|-----------|--------|
| 查询方式 | `WHERE content LIKE '%JVM%'` | cosine(query_vec, memory_vec) |
| 能否匹配同义词 | 不能 | 能（"JVM"匹配"Java虚拟机"） |
| 性能（10 万条） | 全表扫描，慢 | ANN 索引，毫秒级 |
| 模糊程度 | 只支持通配符 | 支持语义模糊度控制 |

## 六、模块 4：去重系统（Deduplication）

### 6.1 为什么需要去重

用户可能反复表达同一个信息：
- 第 1 次："我用 Java"
- 第 3 次："我们项目是 Java 17"
- 第 7 次："技术栈是 Java + Spring Boot"

如果不去重，记忆库会充斥大量重复或高度相似的记忆，浪费存储也影响检索质量。

### 6.2 Mem0 的三层去重策略

```
┌───────────────────────────────────────────────────────────┐
│                    三层去重 Pipeline                        │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  第一层：LLM 语义感知                                      │
│  ┌─────────────────────────────────────────────────────┐ │
│  │ prompt 中传入已有记忆 → LLM 自主判断不重复提取         │ │
│  │ 优点：能理解语义等价（"喜欢Java" ≈ "Java工程师"）     │ │
│  │ 缺点：LLM 不是 100% 可靠，可能漏判                    │ │
│  └─────────────────────────────────────────────────────┘ │
│                         ↓ 漏网之鱼                        │
│  第二层：MD5 Hash 精确匹配                                 │
│  ┌─────────────────────────────────────────────────────┐ │
│  │ MD5(记忆文本) → 与已有 hash + 批次内 hash 对比        │ │
│  │ 优点：零误判，O(1) 查找                               │ │
│  │ 缺点："用Java" 和 "使用Java" hash 不同，无法识别      │ │
│  └─────────────────────────────────────────────────────┘ │
│                         ↓ 漏网之鱼                        │
│  第三层：MySQL 唯一约束兜底                                │
│  ┌─────────────────────────────────────────────────────┐ │
│  │ UNIQUE KEY(user_id, content_hash) → insert 时报错跳过 │ │
│  │ 优点：数据库级别保证，绝对不会重复                     │ │
│  │ 缺点：只能防完全相同的文本                            │ │
│  └─────────────────────────────────────────────────────┘ │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

### 6.3 Mem0 V2 的更新判定机制（UPDATE_MEMORY_PROMPT）

除了去重，Mem0 还有一个「更新判定」机制。当新记忆与旧记忆**相关但不完全相同**时，判断是 ADD、UPDATE 还是 DELETE：

```
已有记忆："喜欢芝士披萨"
新提取事实："喜欢鸡肉披萨"

→ LLM 判断：UPDATE → 合并为"喜欢芝士和鸡肉披萨"
```

```
已有记忆："住在北京"
新提取事实："刚搬到上海"

→ LLM 判断：UPDATE → 更新为"住在上海（从北京搬来）"
```

这个机制在 `configs/prompts.py:176` 的 `DEFAULT_UPDATE_MEMORY_PROMPT` 中定义。

## 七、模块 5：混合检索与评分融合（Hybrid Search + Score Fusion）

这是 Mem0 检索侧最核心、最有技术深度的部分。

### 7.1 为什么需要「混合检索」

单一检索方式各有缺陷：

| 检索方式 | 擅长 | 不擅长 |
|----------|------|--------|
| 语义检索（Embedding） | 理解同义词、模糊匹配 | 精确关键词匹配（如型号"RTX 4090"） |
| 关键词检索（BM25） | 精确匹配特定词 | 无法理解同义词 |
| 实体检索（Entity） | 找到与特定实体相关的记忆 | 覆盖面窄 |

**混合检索 = 三种方式的结果取并集，然后统一打分排序**。

### 7.2 Mem0 混合检索的完整流程（9 步）

```
用户查询："JVM 线程池调优"
      │
      ▼
┌─────────────────────────────────────────────────────────┐
│ Step 1: 查询预处理                                       │
│ ┌─────────────────────────────────────────────────────┐ │
│ │ 词形还原（Lemmatization）：                          │ │
│ │   "attending" → "attend"                            │ │
│ │   "memories" → "memory"                            │ │
│ │ 实体提取：                                          │ │
│ │   "JVM" → (TOPIC, "JVM")                           │ │
│ │   "线程池" → (TOPIC, "线程池")                      │ │
│ └─────────────────────────────────────────────────────┘ │
│                         │                               │
│                         ▼                               │
│ Step 2: 向量化查询                                       │
│   embedding = embed("JVM 线程池调优")                    │
│   → [0.23, -0.45, 0.12, ..., 0.67]                     │
│                         │                               │
│            ┌────────────┼────────────┐                  │
│            ▼            ▼            ▼                  │
│ Step 3: 语义搜索   Step 4: BM25    Step 6: 实体搜索    │
│ ┌──────────────┐ ┌──────────────┐ ┌──────────────────┐ │
│ │ over-fetch   │ │ 关键词匹配   │ │ 从 entity_store  │ │
│ │ topK × 4     │ │ 词频+文档    │ │ 检索实体关联的   │ │
│ │ = 60 条候选  │ │ 频率评分     │ │ 记忆 ID          │ │
│ └──────────────┘ └──────────────┘ └──────────────────┘ │
│            │            │            │                  │
│            ▼            ▼            ▼                  │
│ Step 7-8: 评分融合（Score and Rank）                     │
│ ┌─────────────────────────────────────────────────────┐ │
│ │ combined = (semantic + bm25 + entity) / max_possible│ │
│ │                                                     │ │
│ │ 门控：semantic_score >= 0.1 才进入融合               │ │
│ │ 归一化：max_possible 根据可用信号动态调整            │ │
│ └─────────────────────────────────────────────────────┘ │
│                         │                               │
│                         ▼                               │
│ Step 9: 返回 top-K 结果                                  │
│   取融合分数最高的前 K 条记忆                            │
└─────────────────────────────────────────────────────────┘
```

### 7.3 Over-fetch 策略详解

**问题**：如果只让向量库返回 top-5，可能会漏掉一些语义分数稍低但 BM25 分数很高的记忆。

**解决方案**：请求 4 倍的候选集，在内存中做精排后再截断。

```python
# Mem0 源码 main.py:1597
internal_limit = max(limit * 4, 60)  # 用户要 5 条 → 实际搜索 60 条
semantic_results = self.vector_store.search(query=query, vectors=embeddings, top_k=internal_limit)
```

**为什么是 4 倍？**
- 太少（2x）：可能漏掉关键词匹配强但语义稍弱的记忆
- 太多（10x）：浪费计算资源，绝大多数候选分数很低
- 4x 是经验值，保证 > 95% 的有价值记忆被覆盖

### 7.4 BM25 关键词检索详解

**BM25 是什么？**
BM25 是信息检索领域经典的关键词评分算法（Elasticsearch 的默认算法）。它考虑：
- **词频（TF）**：这个词在文档中出现几次
- **逆文档频率（IDF）**：这个词在所有文档中有多稀有
- **文档长度归一化**：避免长文档天然得高分

**Mem0 中 BM25 的流程：**

```
原始查询 → 词形还原（lemmatize） → 向量库的 keyword_search 接口
                                           │
                                           ▼
                                    得到 raw BM25 scores
                                           │
                                           ▼
                                    Sigmoid 归一化到 [0, 1]
```

**为什么需要归一化？**

BM25 原始分数范围是 [0, +∞)，无法直接和 cosine similarity [0, 1] 相加。Mem0 用 sigmoid 函数映射：

```python
# mem0/utils/scoring.py:43
def normalize_bm25(raw_score, midpoint, steepness):
    return 1.0 / (1.0 + math.exp(-steepness * (raw_score - midpoint)))
```

sigmoid 的 midpoint 和 steepness 根据查询长度动态调整：
- 短查询（≤3 词）：midpoint=5.0, steepness=0.7
- 中查询（4-6 词）：midpoint=7.0, steepness=0.6
- 长查询（7-9 词）：midpoint=9.0, steepness=0.5

### 7.5 实体 Boosting 详解

**什么是实体 Boosting？**

从查询中提取出实体（人名、技术名、项目名），然后在一个独立的 entity_store 中搜索与这些实体关联的记忆，给这些记忆额外加分。

```
查询："Spring Boot 项目的配置"
提取实体：[("TOPIC", "Spring Boot")]
     │
     ▼
entity_store 中搜索 "Spring Boot" 相关的记忆 ID
     │
     ▼
找到：memory_id_7, memory_id_12 与 "Spring Boot" 关联
     │
     ▼
给 memory_7 和 memory_12 加 entity_boost 分数（最高 0.5）
```

### 7.6 评分融合公式

```python
# mem0/utils/scoring.py:60
# 最终分数 = 各信号之和 / 最大可能分数

max_possible = 1.0                    # 基础（语义搜索）
if has_bm25:   max_possible += 1.0    # BM25 贡献最大 1.0
if has_entity: max_possible += 0.5    # 实体 boost 最大 0.5

combined = (semantic_score + bm25_score + entity_boost) / max_possible
```

**示例：**

| 记忆 | 语义分 | BM25 分 | 实体 boost | 融合分 |
|------|--------|---------|-----------|--------|
| "用户使用 Java 17" | 0.85 | 0.3 | 0.0 | (0.85+0.3+0) / 2.5 = 0.46 |
| "JVM 线程池调优经验" | 0.72 | 0.9 | 0.4 | (0.72+0.9+0.4) / 2.5 = 0.81 |
| "喜欢表格对比方案" | 0.90 | 0.0 | 0.0 | (0.90+0+0) / 2.5 = 0.36 |

注意：第二条记忆语义分不是最高，但 BM25（精确匹配"线程池"）+ 实体 boost（JVM 相关）让它排到了第一。这就是混合检索的价值。

### 7.7 门控机制（Threshold Gate）

```python
if semantic_score < threshold:  # 默认 threshold = 0.1
    continue  # 语义分太低，即使 BM25 再高也不要
```

**为什么需要门控？**

防止一条语义完全不相关的记忆，仅仅因为碰巧包含了某个关键词而被推到高位。比如：

- 查询："推荐一家餐厅"
- 记忆："用户在餐厅写了一篇关于 Kubernetes 的文章"

BM25 会因为"餐厅"这个词给高分，但语义上这条记忆和「找餐厅吃饭」完全无关。门控确保语义分必须超过最低阈值。

## 八、模块 6：变更历史审计（History）

### 8.1 为什么需要历史记录

记忆不是一成不变的。用户的偏好会变化：
- 去年喜欢 Java，今年转 Go 了
- 上个月住北京，这个月搬上海了

需要一个审计日志记录**每条记忆的完整生命周期**。

### 8.2 Mem0 的 History 表结构

```python
# mem0/memory/storage.py — SQLiteManager
# history 表记录每一次记忆变更
{
    "memory_id": "uuid-string",
    "old_memory": "之前的内容",      # UPDATE/DELETE 时有值
    "new_memory": "新的内容",        # ADD/UPDATE 时有值
    "event": "ADD | UPDATE | DELETE",
    "created_at": "2026-07-17T10:30:00Z",
    "is_deleted": 0
}
```

### 8.3 History 的价值

1. **可追溯**：知道记忆是什么时候、从什么变成什么的
2. **可回滚**：发现 LLM 抽取错误时可以恢复
3. **可审计**：面试时可以展示「完整的记忆变更生命周期管理」
4. **Debug 友好**：排查"为什么 Agent 给出了错误建议"时，可以追溯当时的记忆状态

## 九、V3 Pipeline 完整流程（写入侧）

将上述模块组合起来，Mem0 V3 的写入流程（`_add_to_vector_store`）：

```
用户完成一次对话
        │
        ▼
Phase 0: 上下文收集
        │  从 SQLite messages 表取该 session 最近 10 条消息
        │  目的：给 LLM 提供上下文，帮助理解代词和指代
        ▼
Phase 1: 已有记忆检索
        │  将新消息 embed → 向量库 search topK=10
        │  目的：告诉 LLM"这些已经记住了，不用重复提取"
        ▼
Phase 2: LLM 抽取（单次调用）
        │  system_prompt + user_prompt（含已有记忆 + 新消息 + 上下文）
        │  response_format = {"type": "json_object"}
        │  返回：{"memory": [{"text": "...", "attributed_to": "..."}]}
        ▼
Phase 3: 批量 Embedding
        │  将抽取出的所有记忆文本一次性向量化
        │  失败则逐条 fallback（容错）
        ▼
Phase 4: Hash 去重
        │  MD5(text) → 与已有 hash + 批次内 hash 对比
        │  重复则跳过
        ▼
Phase 5: 词形还原
        │  为每条记忆生成 lemmatized 文本（供 BM25 使用）
        ▼
Phase 6: 批量持久化
        │  向量库 batch insert + History batch insert
        ▼
Phase 7: 实体链接（可选）
        │  spaCy 提取实体 → entity_store 建立关联
        ▼
Phase 8: 保存原始消息
        │  SQLite messages 表保存，供下次 Phase 0 使用
        ▼
完成 ✓
```

## 十、检索侧完整流程

```
用户发起新的写作请求（或主动搜索记忆）
        │
        ▼
Step 1: 查询预处理
        │  词形还原 + 实体提取
        ▼
Step 2: 向量化查询
        │  embed(query) → query_vector
        ▼
Step 3: 语义搜索（over-fetch 4x）
        │  向量库返回 limit×4 条候选
        ▼
Step 4: BM25 关键词搜索
        │  向量库的 keyword_search 接口
        ▼
Step 5: BM25 归一化
        │  sigmoid 映射到 [0, 1]
        ▼
Step 6: 实体 Boost 计算
        │  从 entity_store 检索关联记忆
        ▼
Step 7: 评分融合
        │  combined = (semantic + bm25 + entity) / max_possible
        │  门控：semantic >= threshold
        ▼
Step 8: 排序 + 截断
        │  按 combined 降序，取 top-K
        ▼
Step 9: 格式化返回
        │  包含 memory_id, content, score, metadata
        ▼
返回给调用方 → 注入到 Agent 的 prompt 中
```

## 十一、总结对比表

| 核心模块 | 做什么 | 用什么方法 | 为什么这么做 |
|---------|--------|-----------|-------------|
| LLM 抽取 | 从对话中提炼结构化记忆 | Prompt Engineering + JSON Output | 只有 LLM 能理解语义和判断价值 |
| Embedding | 文本 → 向量 | DeepSeek/OpenAI Embedding API | 使语义相似度可量化计算 |
| Vector Store | 存储和检索向量 | Qdrant/pgvector/内存 | 支持毫秒级语义搜索 |
| 去重 | 防止重复记忆堆积 | LLM感知 + MD5 + DB唯一约束 | 三层防线，精度递增 |
| 混合检索 | 多路召回 + 评分融合 | 语义 + BM25 + 实体 Boost | 单一方式各有盲区 |
| 变更历史 | 记录记忆生命周期 | SQLite/MySQL 审计表 | 可追溯、可回滚、可审计 |

## 十二、本项目 V1 简化策略

对照 Mem0 完整方案，我们 V1 的策略是**保留核心价值，削减工程复杂度**：

| Mem0 完整方案 | 本项目 V1 | 简化原因 |
|--------------|----------|---------|
| 24 种向量库 | 接口抽象 + 内存实现 | 工厂模式保留，但只需 1 种实现 |
| BM25 关键词搜索 | 跳过 | 中文需分词器（HanLP），复杂度高收益低 |
| spaCy 实体提取 | 跳过 | 中文 NER 效果不稳定，v2 补 |
| 实体 Boost | 跳过 | 依赖实体提取 |
| Reranker | 跳过 | 记忆量小时语义搜索精度够用 |
| 记忆 UPDATE 判定 | 跳过 | v1 只做 ADD，v2 补 UPDATE |
| 语义 + BM25 + 实体融合 | 纯语义 + 时间衰减 | 足够覆盖写作场景 |

**V1 保留的核心：**
- ✅ V3 Pipeline（6 阶段）
- ✅ LLM 抽取 + 三层去重
- ✅ Over-fetch 4x 策略
- ✅ 时间衰减精排（替代 BM25 + Entity 的角色）
- ✅ 变更历史审计
- ✅ 向量存储接口抽象（未来可切换）
